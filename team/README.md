# 个人开发团队 - 基于 Kimi K2.5 的多智能体系统

## 系统架构

这是一个模拟完整软件开发团队的多智能体系统，每个角色由独立的AI会话承担，通过主会话（你）协调。

## 团队角色

### 1. 产品经理 (PM)
- **职责**: 需求分析、功能规划、用户故事编写
- **输出**: PRD文档、功能列表、优先级排序

### 2. 架构师 (Architect)
- **职责**: 技术选型、系统设计、接口定义
- **输出**: 技术方案、架构图、API设计

### 3. 前端开发 (Frontend)
- **职责**: UI实现、交互逻辑、页面组件
- **输出**: 前端代码、组件库、样式文件

### 4. 后端开发 (Backend)
- **职责**: 服务端逻辑、数据库设计、API实现
- **输出**: 后端代码、数据模型、接口文档

### 5. 测试工程师 (QA)
- **职责**: 测试用例、Bug发现、质量保障
- **输出**: 测试计划、测试用例、测试报告

### 6. 运维工程师 (DevOps)
- **职责**: 部署配置、CI/CD、监控告警
- **输出**: 部署脚本、Dockerfile、流水线配置

## 工作流程

```
你(用户) → PM → Architect → Frontend/Backend → QA → DevOps
                ↓              ↓
            技术评审 ← 代码评审
```

## 使用方法

### 启动项目
1. 告诉主会话你的项目想法
2. 主会话会按顺序启动各个角色会话
3. 每个角色完成工作后，结果汇总到主会话
4. 你可以随时介入调整方向

### 日常开发
- **新需求**: 直接告诉主会话，由PM拆解
- **技术问题**: 主会话协调Architect和开发角色
- **Bug修复**: QA定位，开发角色修复
- **部署上线**: DevOps处理

## 角色Prompt设计原则

每个角色都有:
1. **身份定义** - 你是谁
2. **职责范围** - 你做什么/不做什么
3. **工作方式** - 如何与其他角色协作
4. **输出标准** - 交付物格式要求
5. **约束条件** - 技术栈、规范等
